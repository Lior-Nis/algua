## Problem (grounded)

- The backtest simulates on `adj_close` (`bar-schema.md:49`) — a **total-return** series with ordinary cash dividends reinvested retroactively at the **ex-date** (`corpactions.py`: `div_mult = (p_prev - cash) / p_prev` applied at the ex-date boundary).
- The live/paper ledger sizing path is a **price-return** basis: `build_sizing_snapshot` (`live_sizing.py:43-93`) marks positions at **raw** `close` (`_latest_marks` reads `bars["close"]`, the raw column per the schema) and computes `NAV = allocation + Σ position_pnl(fills, mark)` — `position_pnl` (`live_ledger.py:55-83`) is derived **solely from trade fills and the mark**. It never reads the dividend rows in `live_activities` / `paper_venue_activities`.
- `run_tick` (`live_loop.py:268-295`) uses this NAV as **both** the sizing denominator (`equity = min(allocation, NAV)`) **and** the drawdown basis (`check_drawdown(drawdown_equity, peak, max_drawdown)`, peak persisted via `update_nav_peak`).
- **Consequence:** for any dividend-paying long, live/paper NAV drifts below the backtest's total-return equity by the cumulative dividend yield — smaller sized notional, and a drawdown breaker that can halt a healthy book on a phantom drawdown. This corrupts the forward-test certificate, which compares live ticks to the qualifying backtest.

Because **both** the live path (`build_live_sizing_snapshot`) and the multi-tenant paper-venue path (`build_paper_sizing_snapshot`, wired at `paper_cmd.py:418`) go through `build_sizing_snapshot` with raw-close marks, the fix must cover **both lanes** (`LedgerKind.LIVE` and `LedgerKind.PAPER`). Neither lane's mark already embeds dividends, so a dividend accrual credited to NAV is purely **additive** — there is no double-count against an already-total-return mark.

## Decision 0 — the NAV basis definition (resolves GATE-1 finding 5)

**Live/paper ledger NAV is defined as `research-equivalent gross accrued total return`.** It tracks the same accounting basis as `adj_close`:

- Ordinary cash dividends are **accrued gross of withholding tax** at the **ex-date**, matching `adj_close`'s timing and gross basis (`adj_close` reinvests the full declared cash, pre-tax).
- The broker's **payment-date net cash** (post-withholding, post-fees) is a **reconciliation target**, not the NAV input. The withholding/fee gap between the gross accrual and the net cash is an explicitly-tracked **reconciliation residual** routed to suspense — it is NEVER folded into the sizing/drawdown NAV.

**Gross per-share is sourced from corporate-action / dividend-declaration metadata, never derived from account cash (fail-closed).** The NAV accrual's `gross_per_share` MUST come from the declared corporate-action record — the dividend-declaration / corporate-action-announcement metadata that states the **gross, pre-withholding** cash amount per share (plus its stable corp-action id, ex-date, and payment date). The **payment-date account cash is net** (post-withholding/fees) and is deliberately NOT a source for `gross_per_share`: dividing net cash by share count would silently bake the withholding haircut into NAV — exactly the opposite parity break Decision 0 exists to prevent. **If no gross per-share is resolvable from corporate-action metadata for an ex-date event, the event fails closed to suspense** (no accrual is written, NAV is untouched, the operator sees an unresolved-declaration residual). We never fall back to deriving the accrual from net cash received. The declared metadata is the single authoritative gross input; net account cash appears only later, as the payment-date reconciliation target (§2).

**Rationale.** The north star is backtest↔live parity: the frictionless backtest gates candidate promotion and the forward certificate compares live ticks to that backtest. The backtest equity is gross total return (no withholding modeled). If live NAV credited only the net-of-withholding cash, we would trade one parity break (dividends excluded) for the opposite one (dividends under-credited by the tax haircut). Defining NAV on the backtest's own basis makes the sizing denominator and drawdown basis directly comparable to the qualifying backtest. The real withholding cash cost is genuine but belongs in a reconciliation/attribution report, not in the parity-critical NAV.

Everything below follows from Decision 0.

## Decision 0b — one normalized, snapshot-identified declaration contract for BOTH lanes (resolves r2-blockers 1, 2, 7)

Decision 0 says `gross_per_share` comes from a corporate-action declaration. This decision **pins that declaration to a single normalized contract, names where each lane's declarations come from, and states what "same source as the backtest's `adj_close`" can and cannot be proven** — because the earlier draft asserted "the announcements feed" (live) and "the same data the sim applies" (paper) *independently*, which is not parity: parity needs a provable common declaration, or, where that is impossible, a fail-closed reconciliation in its place.

**The contract (pure; importable by execution + tests without crossing the data wall).** Both lanes' accrual consumes exactly one record type. It is a pure dataclass in `algua/contracts/` — no I/O, no `algua.data` import — so the execution lane reads it without breaching the import wall (`corpactions.py` and the CA event data live in `algua.data`, which execution may never import):

```python
# algua/contracts/corpaction_declaration.py   (pure: no I/O, no algua.data)
@dataclass(frozen=True)
class CorporateActionDeclaration:
    symbol: str
    corp_action_id: str        # stable vendor id — the reconciliation join key (§2/§7)
    ex_date: date              # America/NY session date, UTC-midnight rail (#262)
    gross_per_share: float     # DECLARED gross, in RAW-close price units — the SAME units as
                               # corpactions.Dividend.cash (the value that builds adj_close)
    payment_date: date | None
    ca_vendor: str             # NORMALIZED vendor identity — the source-check equality key
                               # (e.g. "alpaca", "firstrate", "yfinance"). NOT a display string.
    source_dataset: str | None # PAPER: the CA-event-manifest snapshot_id the declaration was
                               # materialized from (provable identity). LIVE: the feed name
                               # (e.g. "alpaca_ca_announcements"). Audit/provenance, NOT the gate key.
```

`gross_per_share` is pinned to **RAW-close price units** — identical to `corpactions.Dividend.cash` — so the accrual's per-share figure and the backtest's dividend cash are directly comparable. A units mismatch here would make every "parity" assertion a coincidence; pinning the unit is what makes §5's parity test meaningful.

**Normalized vendor identity vs. display source (resolves IMPORTANT — the earlier `source` string conflated the two).** The source-check gates on **`ca_vendor`** — a normalized vendor token — NOT on the free-form `source_dataset` string. Paper's `source_dataset` is a snapshot id and live's is a feed name; comparing those directly would make live *always* mismatch paper. Instead the strategy's certified `ca_vendor` (the vendor that adjusted the qualifying backtest's `adj_close`, recorded on the certificate — §2/Task 8) is compared to each live declaration's `ca_vendor`; `source_dataset` is retained only for audit. A mismatch (e.g. certified on `firstrate`-adjusted bars but forward-testing on `alpaca` announcements) is a `source_mismatch` discrepancy (§7).

**Reconciliation tolerance constants (resolves IMPORTANT — "≤ the gross per-share" was not a testable rule).** The fallback tuple join and the withholding-residual comparison use two pinned constants, `CA_CASH_RTOL = 1e-6` and `CA_CASH_ATOL = 0.005` (half a cent), applied as `math.isclose(a, b, rel_tol=CA_CASH_RTOL, abs_tol=CA_CASH_ATOL)` for total-cash and per-share comparisons. These are exact, table-testable numbers, not prose.

**A `DeclarationSource` port yields these records; one adapter per lane (r2-blocker 2 — no more "spike then wire or synthesize" hand-wave):**

```python
class DeclarationSource(Protocol):                       # execution-lane Protocol, pure signature
    def declarations_since(self, after_ex_date: date) -> list[CorporateActionDeclaration]: ...
```

- **Live — `AlpacaAnnouncementsSource`.** Wraps a NEW `_AlpacaBroker.corporate_action_announcements(after, until)` read method (Alpaca's corporate-actions **announcements** endpoint), added next to the existing `account_activities_window` (`alpaca_broker.py:411`). This is the execution lane's OWN broker HTTP client — **not** `algua.data` — so the wall holds. Keys declarations to the live account/venue implicitly (one account).
- **Paper — `PaperVenueDeclarationSource`.** SimBroker has **no** declaration surface (confirmed: `sim_broker.py` fills on a price grid; no activities, no CA method). So paper declarations do NOT come from SimBroker at tick time. They are **materialized once, at paper-run provisioning, into a paper-venue-local store** keyed to the venue snapshot, and the source reads that store. The store and its provisioning path are specified in §4.

**What "same source as the backtest's `adj_close`" means — the honest per-lane asymmetry:**

- **Paper — provably identical, but ONLY via a persisted CA-event manifest (resolves HIGH — today there is nothing to replay).** The problem: `corpactions.Dividend` carries only `(ex_date, cash)` — no `corp_action_id`, no vendor — and a bar snapshot persists only the `adj_close` column, NOT the CA events that produced it. So "provisioning writes that exact event list" was not implementable; paper could only *assert* same-source. **Fix — a first-class CA-event manifest as a data-snapshot sidecar** (`algua/data`, non-CODEOWNERS; a parquet/JSON sidecar keyed to the bar `snapshot_id`, like the existing provenance manifest — NOT a registry-db table, so it does not consume the one `db.py` bump): each record `(symbol, corp_action_id, ex_date, gross_per_share, ca_vendor, payment_date | NULL, snapshot_id)` plus a manifest hash, written whenever a snapshot's `adj_close` is built/ingested, and it IS the authoritative event set `corpactions.back_adjust` consumes (so the manifest and `adj_close` cannot drift). `payment_date` is nullable provenance (§4 null-payment policy). At provisioning, the CLI composition root (`paper_cmd.py` — a composition root that MAY legally import `algua.data`; the wall forbids execution *modules* importing `algua.data`, not the CLI that wires the two lanes) reads **the manifest for the paper snapshot** and materializes it into `paper_venue_declarations` with the same `corp_action_id`s and `ca_vendor`. **Fail closed:** a paper snapshot with NO CA-event manifest cannot be dividend-accrued (the run refuses dividend parity rather than replaying an un-provenanced event list). §5's parity test then proves it at runtime by feeding the materialized declarations back through `corpactions.check_adj_close_consistent` against the snapshot's `adj_close`.
- **Live — real-time feed, so identity is impossible; substitute a source-check + fail-closed reconciliation.** A live forward test runs forward in wall-clock time over ex-dates that occur *now* — there is no historical snapshot for those ex-dates to be byte-identical to. "Provably the same data" is therefore replaced by the achievable equivalent, which is fail-closed rather than asserted:
  1. **Source-check (provenance).** The `source`/vendor of the live announcements feed AND the CA-vendor provenance of the qualifying backtest's `adj_close` snapshot are both recorded. A **CA-vendor/derivation mismatch is a discrepancy the forward gate fails closed on** (r2-blocker 6) — an agent cannot silently forward-test on Alpaca announcements against an `adj_close` a *different* vendor adjusted.
  2. **Mechanism reconciliation (the §5 parity test).** Over a historical **overlap** window where both an `adj_close` snapshot and the Alpaca announcements exist, the parity test asserts the accrual's declared `gross_per_share`, fed as a `corpactions.Dividend`, reproduces that snapshot's `adj_close` step within `corpactions`' own tolerance. This proves the live accrual *mechanism* is basis-faithful to `adj_close` on identical inputs.
  3. **Runtime fail-closed inputs.** At tick time the accrual only ever uses a DECLARED gross (never net-derived — Decision 0), and any missing/unmatched declaration is an unresolved-declaration / unmatched-payment discrepancy that fails the window (§2, r2-blocker 6). A divergent or absent live declaration can never pass silently.

  This is the strongest provable statement across a real-time-vs-snapshot seam: not literal identity, but a mechanism proven faithful offline plus a source-check and reconciliation that fail closed at runtime.

**The reconciliation join key — BOTH branches are fully specified so implementation proceeds regardless of the Task-1 spike (resolves HIGH — an under-specified fallback is not implementable).** §2/§7 reconcile the payment-date DIV cash activity to the open accrual. The join has a preferred and a fallback branch; Task 1's spike only selects which is primary, it does not leave the design open:

- **Preferred — strict `corp_action_id` equality**, used iff Task 1 confirms Alpaca's payment-date DIV *account-activity* carries the same `corp_action_id` as the *announcements-feed* declaration. A single exact-id match reconciles; no id or no match → `unmatched_payment` discrepancy (fail-closed).
- **Fallback — deterministic, SIGN-SPLIT tuple join with tolerance**, used iff the ids differ or the activity carries none. `gross_amount` is SIGNED (long accrual > 0, short accrual < 0), and a broker short-dividend debit is negative cash, so a single `abs(net_cash) <= gross_amount` test is wrong (it can never hold for a short and would leave every short obligation a permanent `unmatched_payment` fail). Split by side, matching an unpaid `accrued`/`paid` accrual on `(lane, symbol, payment_date == activity.date)` PLUS:
  - **Long payout** (accrual `gross_amount > 0`, activity `net_cash >= 0`): require `0 <= net_cash <= gross_amount + CA_CASH_ATOL` (withholding only reduces cash) AND `net_cash / entitled_long_qty <= gross_per_share + CA_CASH_ATOL`.
  - **Short debit** (accrual `gross_amount < 0`, activity `net_cash <= 0`): compare `abs(net_cash)` to `abs(gross_amount)` within (`CA_CASH_RTOL`, `CA_CASH_ATOL`), and reconcile against the tenant's `short_obligation` row — NOT the long payout pool.
  **Ambiguity is fail-closed:** if the sign-appropriate predicate matches zero OR more than one open accrual, do NOT reconcile — write an `unmatched_payment` discrepancy for human resolution. Never fuzzy-pick the "closest".

Paper is unaffected either way (the synthetic payment activity carries the store's `corp_action_id`, so the preferred branch always applies). The chosen branch is a single documented predicate in the reconciliation function (Task 7), not a runtime guess.

## Design

### 1. Strategy attribution for non-fill cash — derivation rule, NOT a `strategy_id` column (resolves finding 1)

A dividend cannot carry a `strategy_id` the way `paper_venue_orders` does: the broker account is a **netted custodian** (`live_ledger.py` module docstring — "The broker account is the netted custodian; this ledger is the source of truth for per-strategy attribution"). The broker emits **one account-level** DIV activity per symbol per ex-date; it has no idea which virtual sub-strategy holds those shares. So attribution must be **derived** from the ledger, exactly as position attribution already is.

**Attribution rule.** For a dividend on `symbol` with ex-date `X` and **declared gross per-share `g`** (from corporate-action metadata — Decision 0; if `g` is unresolvable the whole event fails closed to suspense and nothing below runs):

```
strategy_accrual(s)  = g * entitled_qty_at(X, s, symbol)      # signed, per strategy, INDEPENDENT
```

Every strategy is attributed **independently** from the declared gross `g` and its **own** entitled quantity. There is NO division by a shared, signed net-quantity total — so an offsetting long/short book (e.g. tenant A long 100, tenant B short 100, netting to 0) never zeroes out or distorts attribution: A accrues `+100·g`, B accrues `−100·g`, each correct on its own book. (The old design divided the account cash by `total_ledger_qty_at`, a signed sum that collapses to ~0 exactly when longs and shorts offset — a divide-by-near-zero that both blew up the per-share figure and mislabeled a fully-attributable event as "non-attributable." That divisor is gone.)

- `entitled_qty_at(X, s, symbol)` is the strategy's signed position **at the entitlement cutoff** (defined below), reconstructed from the append-only fills ledger: `Σ fills.qty WHERE strategy = s AND symbol = ? AND fill_ts <= entitlement_cutoff(X)`. (The fills table is append-only with `fill_ts`; historical positions are exactly reconstructable — the same sum `believed_positions` computes, with a time filter at the cutoff.)
- **Sign:** a long (`qty > 0`) is **credited** (+), a short (`qty < 0`) is **debited** (−) — a short borrower owes the dividend. `strategy_accrual` inherits the sign of `entitled_qty_at` automatically. Longs and shorts are separately-signed terms in an independent per-strategy computation; they are never summed into a divisor.

**Entitlement cutoff (pinned).** `entitlement_cutoff(X)` is **the close of the last exchange-calendar session strictly before the ex-date `X`**, expressed as a single canonical instant in **one timezone** — the exchange calendar's `America/New_York`, materialized to a UTC instant for storage/comparison against `fill_ts` (which is stored UTC). Concretely: take the exchange trading calendar, find the last session whose date `< X`, take that session's regular close (16:00 `America/New_York`, DST-aware), convert to UTC — that instant is the cutoff. A holder **at that close** carries the position into the ex-date and is entitled (`fill_ts <= cutoff`); a fill after it (i.e. on or after the ex-date session) is **not** entitled. This matches the backtest exactly: `corpactions.py` applies `div_mult` at the ex-date bar boundary, crediting the position established by fills on sessions *before* ex-date; the daily UTC-midnight bar rail (#262) places the prior session close strictly between the prior bar and the ex-date bar, so "held at prior session close" and "held going into the ex-date bar" denote the same position. The cutoff is a single deterministic instant — no wall-clock ambiguity, no per-fill timezone guessing.

- **Extended-hours fills (IMPORTANT — the regular-close assumption is only valid on a regular-hours book).** Using 16:00 regular close as the cutoff is exactly right *iff* fills cannot land between the regular close and the true ex-date entitlement boundary. On the current daily rail the paper/live loop trades on the next-bar reference at session boundaries, so no post-16:00 same-session fill exists and the assumption holds. If extended-hours fills are ever possible (a fill stamped 16:00–20:00 ET on the session before `X`, or an overnight fill before the ex-date session open), a naive 16:00 cutoff would **wrongly exclude** an after-hours holder who is in fact entitled. **This change ships regular-hours-only.** The pinned cutoff is **16:00 ET regular close**, and that is correct because Task 1 confirms the current paper/live loop cannot produce post-16:00 same-session fills. Two guards make this explicit rather than an unstated assumption: (1) Task 3's `resolve_entitlement_cutoff(ex_date, *, session_end)` takes the venue's session-end convention as an explicit parameter rather than hardcoding 16:00 — its ONLY supported value in this change is regular close; (2) **enabling extended-hours trading is out of scope and gated:** it may NOT reuse a naive `20:00 ET` cutoff — the exchange ex-date entitlement boundary is a record-date/settlement convention, so extended-hours support requires its own broker/calendar-backed entitlement rule with its own tests before the `session_end` parameter is given any non-regular value. The parameter exists so that later work is a deliberate, tested change, not a silent entitlement bug — but this issue neither enables nor assumes it.

- **`total_ledger_qty_at(X, symbol)` is retained ONLY for reconciliation, never for NAV attribution.** It is the ledger's net entitled share count and is compared against the broker's declared/paid share count at payment-date reconciliation (§2) to surface orphan/manual holdings and attribution drift. It is not a divisor and not on the NAV path.

**Why derivation, not a column.** A `strategy_id` column on `live_activities`/`paper_venue_activities` would have to be populated by the same ledger derivation anyway (the broker does not supply it), so the column adds a denormalized copy with no new information and a new corruption surface. The derivation reads the one source of truth (fills) directly.

**Non-attributable / orphan residual.** Attribution no longer depends on a nonzero net total, so an offsetting book is fully attributed. The remaining orphan case is a share the **account holds but the ledger attributes to no strategy** (a manual/external holding): the broker's declared/paid share count exceeds the ledger's net long entitled qty. That excess is **not attributable** — its cash is a reconciliation-report residual (surfaces manual holdings / attribution drift) and never touches any strategy NAV. Attribution writes exactly `Σ_s g·entitled_qty_at(X, s, symbol)` across ledger strategies; the orphan gap is visible only at payment-date reconciliation (§2), not in `dividend_accruals`.

### 2. Ex-date accrual + payment-date reconciliation (resolves finding 2)

A dividend is modeled as an **ex-date receivable**, not payment-date cash. Two events, weeks apart:

1. **Ex-date `X`:** the entitlement exists and `adj_close` steps. We accrue the gross receivable into NAV **as of `X`**, independent of when the broker actually pays.
2. **Payment date `P`:** the broker's cash activity lands. It does **not** re-credit NAV (that would double-count the accrual). It **reconciles** against the open accrual: mark it paid, record the net cash received, surface `gross_accrued − net_cash` as the withholding/fee residual.

**New table `dividend_accruals`** (one `db.py` schema bump: `SCHEMA_VERSION 35 -> 36`; the only bump this change makes — see the operating rule on one bump in flight):

```sql
CREATE TABLE IF NOT EXISTS dividend_accruals (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    lane                TEXT NOT NULL,             -- 'live' | 'paper' (LedgerKind.value)
    strategy            TEXT NOT NULL,             -- attributed sub-strategy
    symbol              TEXT NOT NULL,
    corp_action_id      TEXT NOT NULL,             -- stable declaration id — PART OF the uniqueness key
    ex_date             TEXT NOT NULL,             -- ex-date calendar date (America/NY session)
    entitlement_cutoff  TEXT NOT NULL,             -- UTC instant: last session close strictly before ex_date
    gross_per_share     REAL NOT NULL,             -- DECLARED gross (corp-action metadata; never net-derived)
    accrued_qty         REAL NOT NULL,             -- entitled_qty_at(cutoff) (signed)
    gross_amount        REAL NOT NULL,             -- gross_per_share * accrued_qty (signed; NAV input)
    payment_date        TEXT,                      -- broker payment date when known
    status              TEXT NOT NULL,             -- 'accrued' | 'paid' | 'restatement_pending'
    reconciled_cash     REAL,                      -- net cash matched at payment (attributed share)
    restatement_json    TEXT,                      -- corrected declaration held for human review (r2-blocker 4)
    ca_vendor           TEXT NOT NULL,             -- normalized vendor identity (source-check key)
    source_dataset      TEXT,                      -- PAPER: CA-event-manifest snapshot_id; LIVE: feed name (audit)
    source_activity_id  TEXT,                      -- broker activity that drove accrual/payment
    created_ts          TEXT NOT NULL,
    updated_ts          TEXT NOT NULL,             -- bumped on a correcting upsert of an 'accrued' row
    UNIQUE(lane, strategy, symbol, corp_action_id) -- one accrual per DECLARATION, NOT per ex-date
);
CREATE INDEX IF NOT EXISTS ix_dividend_accruals_lane_strategy
    ON dividend_accruals(lane, strategy);
CREATE INDEX IF NOT EXISTS ix_dividend_accruals_corp_action
    ON dividend_accruals(lane, corp_action_id);   -- payment-date reconciliation lookup
CREATE INDEX IF NOT EXISTS ix_dividend_accruals_ex_date
    ON dividend_accruals(lane, strategy, ex_date); -- accrued_dividends(as_of)
```

**Durable discrepancy surface — a new `ca_discrepancies` table the forward gate queries (resolves HIGH: the earlier draft said "no new suspense table; the `*_activities` table is the suspense ledger" — but an unresolved DECLARATION comes from the `DeclarationSource` poll, not a broker activity, so it has NO `*_activities` row, and the §7 gate could not query it. A durable typed table is required):**

```sql
CREATE TABLE IF NOT EXISTS ca_discrepancies (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    lane            TEXT NOT NULL,       -- 'live' | 'paper'
    strategy        TEXT,                -- NULL for account-level (orphan / source-mismatch) rows
    symbol          TEXT NOT NULL,
    corp_action_id  TEXT,                -- NULL when unknown (unmatched-payment w/o id, unresolved-declaration w/o id)
    ex_date         TEXT,                -- session date the discrepancy is attributed to (the gate scans by this)
    kind            TEXT NOT NULL,       -- see kinds below
    gate_failing    INTEGER NOT NULL,    -- 1 = fails the §7 forward gate while open; 0 = informational
    detail_json     TEXT,                -- amounts / net cash / owed / declared-vs-implied / vendor pair
    status          TEXT NOT NULL,       -- 'open' | 'resolved'
    resolved_by     TEXT,                -- human actor on resolution (NULL while open)
    resolved_reason TEXT,
    created_ts      TEXT NOT NULL,
    updated_ts      TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS ix_ca_discrepancies_scan
    ON ca_discrepancies(lane, ex_date, status);   -- the §7 gate window scan
```

- **`kind` and `gate_failing`:** `unresolved_declaration`, `unmatched_payment`, `source_mismatch`, `restatement_pending`, `basis_split_dividend` (§6) are **`gate_failing=1`** — an OPEN row of these kinds within an evidence window fails the §7 forward gate. `withholding_residual`, `short_obligation`, `orphan_residual` are **`gate_failing=0`** — informational reconciliation rows (surfaced in the operator report, never a gate fail; they are expected, not corruption).
- **Single gate surface.** The forward gate (§7) reads **only** `ca_discrepancies` — one durable, queryable table, scanned by `(lane, ex_date ∈ window, status='open', gate_failing=1)`. Every NAV-affecting discrepancy writes a row here. A `restatement_pending` accrual writes BOTH the `dividend_accruals.status` flag (which keeps the gross figure in NAV) AND a `ca_discrepancies(kind='restatement_pending')` row (the gate surface); resolution (§2 controlled path) clears both atomically.
- **Provenance for the source-check (HIGH).** `dividend_accruals.ca_vendor` (normalized) + `source_dataset` (audit) persist WHERE the declaration came from. The qualifying backtest's normalized CA vendor is recorded onto the forward-gate certificate row as `ca_vendor` at mint time (read from the qualified backtest snapshot's CA-event manifest — Decision 0b/Task 8). The live accrual poll compares its declaration `ca_vendor` against the strategy's certified `ca_vendor` (normalized-token equality, NOT the free-form `source_dataset`); a mismatch writes a `source_mismatch` (`gate_failing=1`) row keyed to the affected ex_date. Paper never mismatches (materialized from the same snapshot manifest whose vendor built `adj_close`).

**Numeric convention (MINOR).** Cash and share quantities are `REAL` (float), consistent with the existing ledger (`position_pnl`, `strategy_nav`, `tick_snapshots.equity` are all float); this design does not introduce a mismatched money type. Every reconciliation comparison (withholding residual, sign-split tuple-join per-share and total-cash match, basis parity) uses the pinned `CA_CASH_RTOL = 1e-6` / `CA_CASH_ATOL = 0.005` via `math.isclose`, never exact float equality; Decimal/integer-micro-units for persisted cash is a deferred repo-wide change, not scoped here.

**Uniqueness is per DECLARATION (`corp_action_id`), NOT per ex-date — this is r2-blocker 3.** `corpactions.py` (the backtest's own engine, `algua/data/corpactions.py:85`) explicitly **sums** multiple same-day dividends on one symbol into a single ex-date `div_mult` (`div_cash_sum[ex] = ... + ev.cash`). A `UNIQUE(lane, strategy, symbol, ex_date)` key would give a same-day *second* dividend the same key as the first, so the correction-aware upsert (below) would treat it as a **correction of the first and overwrite it** — silently dropping one of two legitimate same-day dividends and breaking parity with the backtest's sum. Keying uniqueness on `corp_action_id` stores **one row per declaration**; `accrued_dividends(as_of)` then **sums** all rows with `ex_date <= as_of`, which reproduces `corpactions`' same-ex-date cash sum exactly (N declarations on one ex-date → N rows → summed into NAV), and each row still reconciles against its own payment by its own `corp_action_id`. Same-day dividends are aggregated at *read* time by summation, mirroring the backtest — no lossy row-level aggregation, no synthetic aggregate id needed.

**NAV formula change.** `build_sizing_snapshot` (`live_sizing.py`) and `strategy_nav` (`live_ledger.py`) gain one additive term:

```
NAV = allocation
    + Σ_symbol (pnl.realized + pnl.unrealized)          # unchanged, raw-close price return
    + accrued_dividends(conn, strategy, lane, as_of)    # NEW: Σ gross_amount WHERE ex_date <= as_of
```

`accrued_dividends` sums `gross_amount` for the strategy's accruals with `ex_date <= as_of` (rows in `accrued`, `paid`, AND `restatement_pending` status — once accrued the figure stays in NAV until the payment reconciliation or a human-approved correction changes it; the payment-date reconciliation changes `status`/`reconciled_cash`, not the NAV contribution). This is the whole parity fix: NAV now carries the same ex-date-timed total-return dividend contribution as `adj_close`.

**`as_of` is the mark's decision/session date, NOT the tick wall-clock instant (IMPORTANT — precise time semantics).** `accrued_dividends(as_of)` must be dated to the **same session the position marks are dated to** — the decision session the sizing snapshot is being built for — so the dividend contribution and the price marks describe one coherent point in time, exactly as the backtest steps `adj_close` on the ex-date *bar*. Concretely `as_of` is the session `date` derived from the tick's mark timestamp (the bar whose close feeds `_latest_marks`), not `datetime.now()` and not the raw tick submission instant: a tick fired at 09:00 on ex-date `X` marking against the prior session's close must NOT yet include `X`'s accrual (the mark is `X-1`'s close), while the tick marking against `X`'s close does. Comparing a calendar `ex_date` against a calendar session date is a clean date-vs-date comparison (no tz/instant coercion). The mark-session date is threaded from the tick into `build_sizing_snapshot` (Task 9), never re-derived from wall-clock.

**Two independent triggers — accrual is driven by the DECLARATION feed at ex-date, NOT by the payment-date cash activity.** This is the crux of the timing fix and must not be conflated. The broker's DIV **account cash activity typically lands on the payment date `P`, weeks after `X`** — driving the accrual off that activity would leave NAV wrong for the entire `X..P` window (the exact parity break this issue exists to close). So the two events have two separate triggers on two separate feeds:

1. **Accrual trigger — the `DeclarationSource` port (Decision 0b), polled independently every run cycle.** The port carries `CorporateActionDeclaration` records `(symbol, corp_action_id, ex_date, gross_per_share, payment_date, source, snapshot_id)` for declared/upcoming dividends **at or before ex-date**, independent of when cash pays. Each cycle, for every declaration whose `ex_date` has passed since the last poll **and whose `entitlement_cutoff(X) > cutover_ts`** (the single instant-vs-instant cutover predicate — §6; NEVER a calendar `ex_date` vs a timestamp), we compute `entitlement_cutoff` and the per-strategy entitled qty and write the ex-date accrual. Because `accrued_dividends(as_of)` dates the contribution at `ex_date` (`ex_date <= as_of`), NAV reflects the dividend from `X` onward even though the poll that wrote the row runs on/just after `X` — never waiting for `P`.
2. **Reconciliation trigger — the payment-date DIV cash activity.** `_ingest_one_activity` recognizes the whitelisted DIV cash activity and uses it ONLY to reconcile the already-open accrual (match by `corp_action_id`, flip `status='paid'`, record `reconciled_cash`, surface residuals). It NEVER writes or re-credits an accrual — the accrual already exists from trigger 1.

- **Live:** trigger 1 is the `AlpacaAnnouncementsSource` adapter (Decision 0b) over Alpaca's **corporate-actions announcements** endpoint (declared gross cash per share + ex/payment dates + stable `corp_action_id`) — the authoritative gross, pre-withholding declaration, read via the execution lane's own broker client (no `algua.data`). The payment-date DIV account activity (trigger 2, net cash) reconciles it. **Fail-closed on any missing declared input:** a declaration lacking `gross_per_share`, `ex_date`, or `corp_action_id` is not accrued (routed to suspense); we never guess `ex_date = payment_date` (timing break) and never derive `gross_per_share` from net account cash (withholding break). A payment-date DIV cash activity that matches NO open accrual (no announcement was seen) is an **unmatched-payment discrepancy** surfaced to the reconciliation report AND flagged so the forward gate fails the window (§7, r2-blocker 6) — it is NOT back-filled into an accrual from net cash.
- **Paper venue:** trigger 1 is the `PaperVenueDeclarationSource` adapter (Decision 0b) over the `paper_venue_declarations` store, whose concrete interface, key, and provisioning path are specified in §4 (this replaces the earlier "spike then wire or synthesize" hand-wave — r2-blocker 2). Trigger 2 (payment reconciliation) is a synthetic paper DIV cash activity the venue emits at `payment_date` carrying the store's own `corp_action_id`, so the strict `corp_action_id` match is satisfied by construction. Both are materialized from the SAME snapshot CA event list the paper `adj_close` was built from, giving paper provable same-source parity (Decision 0b).

**Accrual insertion is a correction-aware READ-COMPARE-BRANCH, not a single `ON CONFLICT` statement (resolves HIGH — a lone `... DO UPDATE ... WHERE status='accrued'` makes a `paid`-row conflict a silent no-op and can NEVER trigger the restatement path; SQLite gives no signal to branch on).** The write is one function running inside the ingest transaction (`SELECT ... FOR the key`, compare, then branch), NOT a bare upsert:

```
def upsert_accrual(conn, decl, derived_row):
    existing = conn.execute(
        "SELECT status, gross_per_share, accrued_qty FROM dividend_accruals "
        "WHERE lane=? AND strategy=? AND symbol=? AND corp_action_id=?", key).fetchone()
    if existing is None:
        INSERT the new accrual (status='accrued')                       # first sighting
    elif existing.status == 'accrued':
        if (gross_per_share, accrued_qty) unchanged: return              # idempotent no-op, no updated_ts bump
        UPDATE gross_per_share/accrued_qty/gross_amount/entitlement_cutoff, bump updated_ts  # self-heal open row
    elif existing.status in ('paid', 'restatement_pending'):
        if (gross_per_share, accrued_qty) unchanged: return              # identical re-pull of a settled row: no-op
        # a GENUINE post-payment restatement -> controlled correction path (below), NEVER a silent in-place edit
        UPDATE dividend_accruals SET status='restatement_pending', restatement_json=<corrected decl>
        UPSERT ca_discrepancies(kind='restatement_pending', gate_failing=1, status='open', ex_date, corp_action_id)
```

- An **identical re-pull** (of an `accrued`, `paid`, or `restatement_pending` row) changes nothing — idempotent, no `updated_ts` bump. (The key is `corp_action_id`, so a same-day *second* dividend is a distinct declaration → a distinct row, never a false "correction" — r2-blocker 3.)
- A **genuine correction to a still-open (`accrued`) row** overwrites the stale figures and bumps `updated_ts` — the NAV term self-heals to the corrected declaration.
- A **genuine correction arriving after the row is `paid`** (or already `restatement_pending` with a further correction) enters the **controlled correction path** below — the settled `gross_amount` is NOT silently rewritten, but it is NOT left silently wrong forever either (r2-blocker 4).

**Post-payment restatement — a controlled correction path, not silent permanence (r2-blocker 4).** The earlier draft merely "surfaced a discrepancy" while a stale `gross_amount` sat in NAV indefinitely — a corrupt figure could ride the sizing/drawdown NAV and a forward certificate forever with only a report line. That is not fail-closed. Instead, when a genuine correction (a changed declared `gross_per_share`/`accrued_qty` for the same `corp_action_id`) arrives on a `paid`/`restatement_pending` row:

1. **Flag, don't silently rewrite.** In one transaction the row's `status` flips `paid -> restatement_pending`, the corrected declaration is stored in `restatement_json`, AND an OPEN `ca_discrepancies(kind='restatement_pending', gate_failing=1, ex_date, corp_action_id, strategy)` row is written. The live `gross_amount` is deliberately left unchanged for now (a silent mid-flight NAV edit is itself a hazard), but the event is now a durable **unresolved NAV-affecting discrepancy** the gate can see.
2. **The forward gate fails closed on it.** The OPEN `restatement_pending` discrepancy, scanned by `ex_date` within an evidence window, fails the §7 forward gate (r2-blocker 6). Any certificate whose evidence window covers that ex_date is invalidated on re-evaluation, and promotion / forward evidence is BLOCKED while the row is open. The wrong figure can no longer mint or sustain a valid certificate.
3. **Resolution is human-only and audit-logged.** A new CLI action (`--actor human`) resolves it, in one transaction, one of two ways: (a) **apply the correction** — write the delta so NAV moves to the corrected figure (recorded as a correcting adjustment with an audit link to the restated row), set the accrual `status='paid'`, clear `restatement_json`, and set the `ca_discrepancies` row `status='resolved'` with `resolved_by`/`resolved_reason`; or (b) **accept as immaterial** — record a human reason, set `status='paid'`, clear the flag with NO NAV change, resolve the discrepancy row. Both clear the OPEN discrepancy so the gate can pass again; an agent actor is rejected.

This mirrors — and hardens — the existing `activity_id`-dedup discipline in `ingest_activities`: dedup on identity, but let a corrected upstream record correct the derived row (in place while open; via the controlled human path once settled) rather than freezing the first-seen (possibly wrong) value or freezing a wrong post-payment value forever.

### 3. Whitelist of NAV-eligible activity types; everything else fails closed to suspense (resolves finding 3)

Only **ordinary cash dividends** are eligible to touch NAV. A hardcoded allowlist gates it:

```python
NAV_ELIGIBLE_ACTIVITY_TYPES = frozenset({"DIV"})   # ordinary cash dividend only
```

- **Interest (`INT`)** is deliberately **excluded**: interest on a cash balance is an account-level artifact, not a total-return component of any held asset, and it is not in `adj_close`. Crediting it would create a fresh parity break. (Documented as an explicit non-inclusion, not an oversight.)
- **Special / return-of-capital / liquidating distributions** are excluded to stay symmetric with the backtest, which *rejects* them (`corpactions.py` raises when `cash >= p_prev`). A DIV whose declared `gross_per_share >= prior close`, or a non-ordinary DIV subtype (e.g. `DIVROC`, `DIVCGL`, `DIVCGS`, `DIVNRA`), routes to suspense.
- **All other cash** — deposits (`CSD`), withdrawals (`CSW`), journals (`JNLC`/`JNLS`), transfers (`ACATC`/`ACATS`), fees (`FEE`), and any **unrecognized/unknown type** — routes to suspense and can never reach sizing or drawdown. This is the fail-closed default: an activity type must be *explicitly* whitelisted to affect NAV.

**Suspense — two surfaces, one guarantee.** Non-eligible **broker cash** (deposits/withdrawals/journals/fees/interest/unknown) is recorded, as today, in the `*_activities` table, which is **already excluded from NAV** (`position_pnl` / `accrued_dividends` never read it). But a **CA-specific** problem — a whitelisted DIV that failed closed because its declared gross/ex_date/corp_action_id did not resolve — has NO `*_activities` row to live in (it came from the `DeclarationSource` poll, not a broker activity). Those go to the durable **`ca_discrepancies`** table (§2, `kind='unresolved_declaration'`, `gate_failing=1`) so the §7 gate can query them by ex_date. This corrects the earlier draft's claim that "no new suspense table is required": broker cash suspense stays in `*_activities`, but CA discrepancies need the typed `ca_discrepancies` table. The invariant is unchanged: NAV's dividend term is sourced ONLY from `dividend_accruals` (whitelisted, attributed, ex-date-accrued), never from a raw `*_activities` sum and never from an unresolved-declaration row.

### 4. Paper lane — the declaration store, its provisioning, and per-`strategy` accrual (resolves finding 4 [round-1] + r2-blocker 2)

**Where paper declarations come from — the concrete interface (r2-blocker 2).** SimBroker has no declaration surface, so the earlier "spike, then wire or synthesize" is replaced with a designed store + provisioning path:

```sql
CREATE TABLE IF NOT EXISTS paper_venue_declarations (
    venue_snapshot_id   TEXT NOT NULL,   -- keys declarations to the paper snapshot/venue run
    symbol              TEXT NOT NULL,
    corp_action_id      TEXT NOT NULL,   -- stable id, reused by the synthetic payment activity
    ex_date             TEXT NOT NULL,   -- America/NY session date (UTC-midnight rail)
    gross_per_share     REAL NOT NULL,   -- RAW-close units, == corpactions.Dividend.cash
    ca_vendor           TEXT NOT NULL,   -- normalized vendor (so the source yields a full contract)
    payment_date        TEXT,            -- from the manifest; NULL policy below
    snapshot_id         TEXT NOT NULL,   -- the bar snapshot whose adj_close was built from this event
    created_ts          TEXT NOT NULL,
    UNIQUE(venue_snapshot_id, symbol, corp_action_id)
);
```

`ca_vendor` is carried so `PaperVenueDeclarationSource` can reconstruct the full `CorporateActionDeclaration` (finding — the source-check needs it; `dividend_accruals.ca_vendor` is `NOT NULL`). When the source writes the accrual it sets `source_dataset = snapshot_id`.

**Null-`payment_date` policy (finding).** The CA-event manifest carries `payment_date` as **nullable** provenance (a historical snapshot may know the ex-date but not the pay-date). If a paper declaration's `payment_date` is NULL, the venue emits **no** synthetic payment activity: the accrual stays `status='accrued'` (still fully in NAV — the parity fix does not depend on payment), and the "synthetic payment reconciles by `corp_action_id`" acceptance test applies ONLY to declarations with a non-null `payment_date`. A NULL pay-date is therefore not an `unmatched_payment` discrepancy — it is a no-payment-expected accrual.

- **Provisioning (wall-safe).** At paper-run setup, the CLI composition root (`paper_cmd.py`) — which MAY import `algua.data` (the wall forbids execution *modules* crossing to `algua.data`, not the CLI that wires the two lanes) — reads the **CA-event manifest sidecar for the paper snapshot** (Decision 0b — the authoritative `(symbol, corp_action_id, ex_date, gross_per_share, ca_vendor, snapshot_id)` set that `adj_close` was built from) and writes it into `paper_venue_declarations` keyed by `venue_snapshot_id`. A one-time data→paper hand-off OUTSIDE the tick loop. **Fail closed:** a paper snapshot without a CA-event manifest is not dividend-accrued (the run refuses parity rather than replaying an un-provenanced event list).
- **Reading (wall-safe).** At tick time, `PaperVenueDeclarationSource.declarations_since(after_ex_date)` (Decision 0b) reads ONLY `paper_venue_declarations` for the active `venue_snapshot_id` — an execution-lane read of an execution-lane table. The tick loop never touches `algua.data`. This is how the execution lane reads declarations without crossing the data-wall boundary.
- **Payment reconciliation.** The venue emits a synthetic paper DIV cash activity at `payment_date` carrying the store's `corp_action_id`, so §7's strict `corp_action_id` match holds by construction (r2-blocker 7 is a non-issue for paper).
- **Provable same-source parity.** Because the store is materialized from the SAME snapshot `Dividend` list that built `adj_close` (same `snapshot_id`), the paper accrual is provably on the same declaration data as the backtest (Decision 0b).

**Multi-tenancy — per-`strategy` accrual (resolves finding 4).** A `LedgerKind.PAPER` DIV for `AAPL` is **one account-level event** even when three tenants hold `AAPL`. The §1 derivation rule handles this natively and **independently per tenant**: each tenant `s` accrues `g · entitled_qty_at(X, s, AAPL)` from the same declared gross `g`, producing one `dividend_accruals` row **per tenant strategy** (keyed `(paper, s, AAPL, corp_action_id)`). Because each row is computed from `g` and that tenant's own entitled qty — with **no shared divisor** — offsetting tenant books never interfere: a tenant long `AAPL` and a tenant short `AAPL` in the same venue accrue a credit and a debit respectively, both correct, even if the venue nets flat. `accrued_dividends(conn, strategy, 'paper', as_of)` reads only that strategy's rows, so each tenant's NAV, sizing, and drawdown see only their own dividend share. This is the paper analog of how `believed_positions` already scopes fills per `strategy` in the shared `paper_venue_fills` table.

**Multi-tenant payment-cash allocation mechanics + durable reconciliation rows (IMPORTANT — previously undefined).** The *accrual* into NAV never needs allocation — each tenant's `gross_amount` is computed independently. Allocation matters only for the *reconciliation* records, where ONE account-level net cash payment `C_net` for a `corp_action_id` is decomposed into durable `ca_discrepancies` rows (`gate_failing=0`, informational) so nothing is left implicit:

- **Long payout split** (`kind='withholding_residual'` per long tenant): allocate `C_net` to each entitled long tenant **pro-rata by that tenant's share of the account's total entitled LONG qty** — `reconciled_cash(s) = C_net · long_qty(s) / Σ_t long_qty(t)` — and record that tenant's `gross_amount − reconciled_cash(s)` withholding residual. `reconciled_cash(s)` is written to the tenant's `dividend_accruals` row; it is a reconciliation figure ONLY and never re-enters NAV.
- **Short obligations** (`kind='short_obligation'` per short tenant): a short tenant was DEBITED at accrual (NAV already carries `−g·|qty|`); at payment it OWES the dividend and is excluded from the payout split. The owed cash is a durable reconciliation row (owed = `g·|qty|`), settled against the broker's short-dividend debit — not allocated from `C_net`.
- **Orphan residual** (`kind='orphan_residual'`, `strategy=NULL`): any account-vs-ledger share gap (broker paid on more/fewer shares than the ledger's net entitled long qty — a manual/external holding) is one account-level row per event, never allocated to a strategy.

All three are `gate_failing=0` (expected reconciliation, not corruption). Only a genuinely unresolvable payment (no matching accrual at all) is the `gate_failing=1` `unmatched_payment` kind.

### 5. Parity test asserting Decision 0, plus the edge cases (resolves finding 5)

Tests assert the NAV basis is **gross accrued total return** and that it matches the backtest's `adj_close` basis:

> **Data-wall note for these tests.** The parity tests import `algua.data.corpactions` (e.g. `check_adj_close_consistent`) — that is allowed because they are **tests**, and the ONLY production place `algua.data` is touched is the paper-provisioning CLI composition root (§4), never an execution module. Task 10 adds an **import-linter regression test** asserting no `algua.execution` / `algua.contracts` module imports `algua.data`, so the new accrual code cannot quietly grow a wall breach.

1. **Basis parity (the core assertion).** A single long held across an ex-date: the ledger NAV growth from `T-1` to ex-date `X` (raw-close price return **+** the gross dividend accrual) equals the backtest total-return equity growth on the same `adj_close` bars over the same interval, within tolerance. This is the assertion the issue asked for, pinned to the **gross** definition.
2. **Ex-date vs payment-date separation.** Accrual hits NAV at ex-date `X`; the later payment-date cash activity does **not** change NAV (no double-count), only flips `status` to `paid` and records `reconciled_cash`. NAV between `X` and `P` already reflects the dividend.
3. **Multi-strategy same-symbol holding.** Two tenants holding the same symbol across one account-level DIV each accrue `g · own_entitled_qty`; neither sees the other's share; the two gross accruals sum to `g · (qty_A + qty_B)`.
4. **Offsetting long/short book does not zero out (finding 2).** Tenant A long `N` and tenant B short `N` of the same symbol across one DIV (venue nets flat): A accrues `+g·N`, B accrues `−g·N`, both nonzero and correct; no divide-by-near-zero, and the event is NOT mislabeled non-attributable despite the zero net total. Asserts attribution is independent per strategy, not divided by a signed net total.
5. **Short-position debit sign.** A short across the ex-date is **debited** (`gross_amount < 0`), lowering NAV — matching `adj_close` (a short's total return pays the dividend).
6. **Entitlement-cutoff boundary (finding 3).** For an ex-date `X` with a pinned `entitlement_cutoff` (last `America/NY` session close strictly before `X`, as a UTC instant): a buy filled **just before** the cutoff is entitled (accrues); a buy filled **exactly at** the cutoff instant is entitled (`fill_ts <= cutoff`); a buy filled **just after** the cutoff / on the ex-date session is **not** entitled (no accrual). A sell just before the cutoff removes entitlement. Table-driven over the three boundary offsets; asserts a DST-affected ex-date resolves the close correctly in the single canonical timezone.
7. **Gross sourced from declaration, fail-closed on net (finding 1).** A DIV whose corporate-action metadata supplies a declared `gross_per_share` accrues `g·qty`; a DIV with **no resolvable declared gross** (or missing `ex_date`/`corp_action_id`) produces **no** accrual and routes to suspense — the accrual is NEVER derived from the net payment cash. Assert that given a net cash strictly below `g·qty`, NAV still reflects the **gross** `g·qty`, never the net-implied per-share.
8. **Rejection of non-attributable / non-whitelisted cash.** A deposit / withdrawal / journal / fee / interest activity and a DIV on a symbol with zero ledger entitled qty produce **no** NAV change (routed to suspense); an unknown activity type is fail-closed (no NAV change).
9. **Withholding residual + reconciliation key.** When net payment cash < gross accrual, NAV keeps the **gross** accrual and the residual is surfaced in the reconciliation report, not deducted from NAV. The payment activity is matched to its open accrual by **`corp_action_id`** (not a fuzzy date window): a mismatched/absent `corp_action_id` leaves the accrual open and surfaces an unmatched-payment discrepancy.
10. **Idempotency + correction upsert.** Re-ingesting the same DIV window does not double-accrue and does not bump `updated_ts` (upsert `WHERE` predicate is false). A **corrected declaration on a still-open (`accrued`) row** overwrites `gross_per_share`/`accrued_qty`/`gross_amount` and bumps `updated_ts` (NAV self-heals).
11. **Same-day multiple dividends aggregate, do not overwrite (r2-blocker 3).** Two DIV declarations on the SAME `(symbol, ex_date)` with distinct `corp_action_id`s produce **two** `dividend_accruals` rows; `accrued_dividends(as_of)` **sums** both into NAV (matching `corpactions.py` summing same-day cash), and neither is treated as a "correction" of the other. Assert a `UNIQUE(...,ex_date)` key would have collapsed them (regression guard on the key choice).
12. **Post-payment restatement — controlled correction path (r2-blocker 4).** A corrected declaration arriving after a row is `paid` does NOT silently rewrite the settled row; it flips `status='paid' -> 'restatement_pending'`, stores `restatement_json`, and the row becomes an unresolved discrepancy. Assert: (a) the stale `gross_amount` is NOT silently changed; (b) the forward gate now FAILS a window covering that ex_date (item 14); (c) a human resolution — apply-correction (NAV moves to corrected figure, flag cleared) OR accept-immaterial (reason recorded, flag cleared) — clears the discrepancy and is rejected for a non-human actor.
13. **Accounting-basis equality gate (r2-blocker 5).** A certificate/evidence window whose ticks carry an older `accounting_basis_version` than current is INELIGIBLE (`basis_consistent` check fails closed); a window straddling the cutover (mixed-basis ticks) fails; a clean single-current-basis window passes the basis check. `verify_forward_certificate` rejects an old-basis certificate.
14. **Unresolved-CA-discrepancy gate (r2-blocker 6).** An evidence window with an OPEN `gate_failing=1` `ca_discrepancies` row (`unmatched_payment` / `unresolved_declaration` / `source_mismatch` / `restatement_pending`) whose `ex_date` ∈ `[first, last]` observation session FAILS the forward gate (`no_unresolved_ca_discrepancies`); a `gate_failing=0` informational row (withholding/short/orphan) does NOT fail; a resolved row does NOT fail; a clean window passes.
15. **Certificate re-check uses `ex_date <= last_observation_session`, not wall-clock (§7 HIGH).** A `restatement_pending` row recorded AFTER `created_at` whose `ex_date` is BEFORE the first tick but is still summed into every in-window NAV invalidates the certificate in `verify_forward_certificate`; a discrepancy whose ex_date is AFTER the last observation session does not.
16. **Live source-check on normalized vendor + sign-split fallback join (Decision 0b, r2-blockers 1/7).** A live accrual whose declaration `ca_vendor` differs from the certified `ca_vendor` writes a `source_mismatch` row and fails the window (item 14); a matching vendor passes even when `source_dataset` strings differ (live feed name vs snapshot id). Payment reconciliation: strict-`corp_action_id` match reconciles; the sign-split fallback reconciles a unique long payout (`0<=net<=gross+atol`) AND a unique short debit (`abs(net)≈abs(gross)`) against its short-obligation; a payment matching zero OR ≥2 open accruals fails closed to `unmatched_payment` (no fuzzy pick). The mechanism-reconciliation parity test (item 1) runs over a historical overlap window using the declared dividends fed through `corpactions.check_adj_close_consistent` (test-only import).
17. **Basis-split dividend fails closed (§6, cutover parity hole).** A dividend with `entitlement_cutoff <= cutover_ts < ex_date-mark-session` is NOT accrued (correct — no retroactive step) but writes a `basis_split_dividend` (`gate_failing=1`) row at cutover; an evidence window reaching that ex_date FAILS the gate; a human accept-immaterial resolution clears it and the window then passes; a window entirely before that ex_date is unaffected.

### 6. Rollout policy for the retroactive NAV step-change (resolves finding 6)

The fix raises NAV for every dividend-paying long. Applied retroactively to a running book it would step NAV up (and, if peaks ratchet, could **mask** a subsequent real drawdown) — or, if peaks were rebased down, could **trip** one. Chosen policy: **prospective-only accrual from a per-strategy cutover, plus an accounting-basis stamp; no peak rebase.**

- **Prospective cutover — gated on the entitlement INSTANT, not a date-vs-timestamp compare.** `cutover_ts` is the per-strategy deploy **instant** (UTC), persisted on first tick under the new basis. The accrual filter is `entitlement_cutoff(X) > cutover_ts` — **instant vs instant** (both UTC; `entitlement_cutoff` is the pinned §1 last-session-close instant). We deliberately do NOT compare the calendar `ex_date` against a timestamp: that is a type/semantics mismatch, and it would wrongly accrue a dividend whose entitlement crystallized **before** the cutover but whose ex-date session merely happens to fall on/after the deploy day (e.g. deploy at 10:00 on an ex-date whose entitlement cutoff was the prior session close — pre-cutover). Gating on `entitlement_cutoff > cutover_ts` guarantees the entire economic event (the moment you had to hold to be entitled) arose under the new basis, so there is **no retroactive NAV step** — nothing to rebase, no phantom trip/mask on deploy. Any dividend whose entitlement crystallized at or before the cutover stays excluded (it was also excluded from the peak that gated the running book, so the book stays internally consistent). **Boundary policy (explicit):** the predicate is STRICT `>`, so an event whose `entitlement_cutoff == cutover_ts` exactly is **excluded** (treated as pre-cutover). This is deliberate: `cutover_ts` is the deploy instant persisted on first tick, an entitlement crystallizing at that same instant belongs to the pre-cutover book (whose peak did not include it), so excluding it keeps the running book internally consistent. Task 8 tests the `==` boundary explicitly.

- **Basis-split dividends fail closed, they are NOT silently shipped (resolves HIGH — the residual parity hole in prospective rollout).** There is a narrow class the entitlement-cutoff gate excludes but `adj_close` includes: a dividend with `entitlement_cutoff <= cutover_ts < ex_date-mark-session` — entitlement crystallized pre-cutover (so, correctly, NOT accrued, to avoid a retroactive step and stay consistent with the pre-cutover peak), yet its `adj_close` step lands on a post-cutover session. Live NAV omits `D`; the gross-basis curve `adj_close` carries `+D` from that ex-date onward, and because the dividend is additive-not-multiplicative the omission perturbs **every** post-split return, not just one — so the live curve is not a perfectly faithful gross curve over any window reaching that ex-date. This is inherent to a no-retroactive-step rollout; the fix is to make it **visible and human-gated, never silent.** At cutover the poll enumerates every declared dividend with `entitlement_cutoff <= cutover_ts < ex_date-mark-session` for the strategy's holdings and writes a `ca_discrepancies(kind='basis_split_dividend', gate_failing=1, ex_date=<mark session>, strategy)` row. The §7 gate scan (`ex_date <= last_observation_session`, gate_failing=1) then fails any evidence window whose observations reach that mark session — so a certificate can NOT be earned on a basis-split-contaminated curve. A human resolves it (§4 resolve CLI): **accept-immaterial** after confirming `D` is negligible (the common case — a single sub-percent dividend), or re-baseline. This reuses the existing discrepancy machinery, honors the entitlement-cutoff predicate (§IMPORTANT: we do NOT flip back to an `ex_date`-vs-timestamp gate), and closes the parity hole fail-closed rather than accepting a slightly-off certificate. Task 8 tests the `entitlement_cutoff <= cutover_ts < ex_date` case produces a `basis_split_dividend` row that fails the window until human-resolved.
- **Accounting-basis version stamp + a REQUIRED basis-equality gate check (r2-blocker 5 — a committed protected-path change, not a spike).** The earlier draft deferred the decision to a spike ("if it can be satisfied in a non-protected read path, preserve auto-merge"). Grounding in the code closes that: `tick_snapshots` has **no** accounting-basis column and `forward_promotion.assemble_forward_evidence` (which `SELECT`s `id, tick_ts, decision_ts, equity, ... FROM tick_snapshots`) and `forward_gate_evaluations`/certificate reads carry **no** basis field. There is no non-protected read path that already sees the basis, so the check REQUIRES a real schema + protected read-path change. The committed design:
  - **Schema (Task 2, non-protected `db.py`):** add `accounting_basis_version INTEGER` to `tick_snapshots` and to the forward-gate evaluation (certificate) row, plus the `ACCOUNTING_BASIS_VERSION` constant (bumped to mark the dividend-accrual basis).
  - **Write path:** every NAV tick stamps the basis version it was computed under; `run_forward_gate` records the basis on the certificate row.
  - **Protected read path (Task 8 — `forward_promotion.py` and `forward_gates.py`, CODEOWNERS):** `assemble_forward_evidence` adds `accounting_basis_version` to its `tick_snapshots` `SELECT` and derives `basis_ok = (every admissible tick's basis == the current ACCOUNTING_BASIS_VERSION)` — a window that straddles the cutover (any admissible tick under an older basis, i.e. mixed-basis observations) sets `basis_ok=False`. `ForwardEvidence` gains a `basis_ok: bool` field; `evaluate_forward_gate` (pure) adds a `basis_consistent` check that fails closed when `basis_ok` is False. `verify_forward_certificate` additionally rejects (fail-closed `TransitionError`) a certificate whose recorded `accounting_basis_version` is older than the current basis. **Explicit rule:** evidence measured under an older `accounting_basis_version` than the strategy's current live basis — or a window straddling the cutover — is INELIGIBLE to satisfy any post-cutover parity/promotion comparison; the gate treats it as no valid certificate, never silently comparing pre-cutover (old-basis) evidence against a post-cutover requirement. A strategy seeking a fresh certificate re-runs the paper→forward window from the cutover forward (clean, single-basis evidence). **This touches CODEOWNERS-protected files (`forward_gates.py`, `forward_promotion.py`); the PR therefore stays OPEN for human merge (no auto-merge) — accepted, not deferred.**
- **No peak rebase.** Because accrual is prospective, `live_nav_peaks` / `strategy_peaks` need no migration — the peak simply begins ratcheting on the new (dividend-inclusive) NAV from the cutover, and the first post-cutover ex-date raises NAV and peak together (no drawdown artifact).
- **Explicitly rejected alternatives:** (a) retroactive back-credit since inception + peak rebase — larger blast radius, must touch peak tables, and any rebase error directly arms/masks the drawdown breaker; (b) apply retroactively but leave peaks — guarantees a mask or a trip on deploy. Prospective + basis-stamp is the minimal, safe path and keeps the drawdown breaker honest.

### 7. Forward-gate criterion — fail any window with an unresolved NAV-affecting CA discrepancy (resolves r2-blocker 6)

Today the forward gate treats dividend cash as harmless noise: `forward_promotion.py:45` comments *"DIV/INT/FEE are position-attributable and pass"*, and `_classify_activities` only counts external-capital types and unattributable fills. So a window whose NAV is silently corrupted by a dividend problem — a payment with no matching accrual (unmatched-payment), a declaration that failed closed to suspense (unresolved-declaration), a CA-vendor/source mismatch vs the qualifying backtest (Decision 0b), or a post-payment restatement (`restatement_pending`, §2) — could still mint a PASS certificate. The certificate would then be built on a wrong equity curve. This adds the missing gate wall:

- **Discrepancy surface the gate reads — the ONE `ca_discrepancies` table (§2).** All unresolved NAV-affecting kinds live in one durable, typed, indexed table: `restatement_pending`, `unresolved_declaration`, `unmatched_payment`, `source_mismatch` (all `gate_failing=1`). No dependency on broker-activity rows (an unresolved declaration has none) and no multi-table union — the earlier draft's "the `*_activities` table is the suspense ledger" could not express this, which is why §2 adds the table.
- **Scan bound is `ex_date <= last_observation_session`, NOT `BETWEEN first AND last` (resolves HIGH — a lower bound leaks pre-window contamination).** `accrued_dividends(as_of)` sums EVERY accrual with `ex_date <= as_of`, so an accrual whose `ex_date` is BEFORE the first observation session is STILL summed into every in-window NAV tick (and the sizing denominator, and the drawdown series). A `BETWEEN first AND last` scan would miss a bad/restated pre-window accrual that contaminates the entire certified period. The correct scan is every OPEN `gate_failing=1` row whose accrual can affect an in-window tick: `SELECT COUNT(*) FROM ca_discrepancies WHERE lane=? AND (strategy=? OR strategy IS NULL) AND status='open' AND gate_failing=1 AND ex_date <= ?last_observation_session`. (All accrual rows are post-cutover by construction — the accrual filter only writes `entitlement_cutoff > cutover_ts` rows — so no lower bound is needed; pre-cutover events are simply never accrued.)
- **Evidence field + pure check (protected files — Task 8).** `assemble_forward_evidence` counts OPEN `gate_failing=1` `ca_discrepancies` with `ex_date <= last_observation_session` for this strategy and lane (account-level `strategy IS NULL` rows count too); `ForwardEvidence` gains `n_unresolved_ca_discrepancies: int`; `evaluate_forward_gate` adds a `no_unresolved_ca_discrepancies` boolean check that fails closed when the count is non-zero (actionable detail).
- **Certificate re-check over `ex_date <= last_observation_session`, not `[created_at, now]` (resolves HIGH — a wall-clock re-check is the wrong domain).** A restatement for an ex_date whose accrual is in the certified NAV can arrive AFTER certification, and its `ex_date` may be BEFORE `created_at` or even before the first tick — a `[created_at, now]` scan would miss it. The certificate row already stores `last_tick_ts` (the window's end anchor). `verify_forward_certificate` derives the last observation **session** from it and re-scans `ca_discrepancies` by `ex_date <= last_observation_session` (plus the basis-version check), so any discrepancy — whenever recorded, whatever its ex_date, as long as it feeds an in-window NAV — fails the live-wall re-verification. This is the ex_date-domain analog of its existing reconcile/kill-trip re-checks.
- **Interaction with §2 and §6.** This is the enforcement arm of r2-blocker 4's controlled correction path (a `restatement_pending` row blocks promotion until a human resolves it) and of Decision 0b's live source-check (a vendor mismatch blocks the window). It is a NEW gate criterion, additive to the existing checks; it never relaxes an existing one.
- **CODEOWNERS.** Lands in `forward_gates.py` + `forward_promotion.py` (same protected files as r2-blocker 5) — PR stays OPEN for human merge.
